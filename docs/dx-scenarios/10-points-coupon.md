# DX Scenario 10: ポイント + クーポン

## アプリ概要

ポイント付与・残高管理・クーポン適用・履歴を備えたロイヤリティ API。

| 機能 | エンドポイント例 |
|------|----------------|
| ポイント付与 | `POST /users/{id}/points`（amount, reason, expires_at） |
| ポイント利用 | `POST /users/{id}/points/use`（amount, order_id） |
| ポイント残高 | `GET /users/{id}/points/balance` |
| ポイント履歴 | `GET /users/{id}/points/history?page=1` |
| クーポン管理 | `POST /coupons`（code, discount_type, value, min_amount, valid_until） |
| クーポン検証 | `GET /coupons/{code}/validate?order_amount=5000` |
| クーポン適用 | `POST /orders/{id}/apply-coupon`（coupon_code） |
| クーポン使用履歴 | `GET /coupons/{code}/usage` |

ポイント: 有効期限付きポイントの残高計算（FIFO 消費）、クーポン重複利用防止、割引計算。

---

## Persona A — 坂本 ひかる（新卒・女性・24 歳）

### 背景

ECデザイン専攻の専門学校卒でエンジニアに転向。ポイントカードは使ったことがあるが実装は未経験。

### 作業シナリオ

1. `users` テーブルに `point_balance INTEGER` カラムを直接追加する設計を選ぶ。
2. ポイント付与は `point_balance += amount` のシンプルな UPDATE。
3. 有効期限付きポイントの概念に気づかず、全ポイントが永久有効という設計になる。
4. クーポンは `coupons` テーブルを作るが、「1 人 1 回まで」の制限実装を忘れる。
5. 割引計算（定額/パーセンテージ）を `if/else` でコントローラーに書いてしまう。

### ハマりポイント

- **ポイント残高の管理方法**: `point_balance` カラムの直接更新 vs 履歴テーブルからの計算の選択。
- **有効期限付きポイント**: FIFO 消費（古いポイントから使う）の複雑さに気づかない。
- **クーポン一人一回制限**: `coupon_usages(coupon_id, user_id)` テーブルの設計が必要。

### 解決策 & 感想

先輩に「ポイントは履歴テーブルで管理する」を教わり、`point_ledger` テーブルに設計変更。
有効期限は「有効分のみ SUM する SQL」に改修した。

> 「ポイントって残高カラムで持つのが普通だと思ってた。
>  でも履歴で持てば何かあったとき追跡できるって言われて確かになと思った。
>  有効期限の計算は難しかった。」

### DX スコア: ⭐⭐（2/5）

有効期限設計で大幅改修必要。ポイント設計パターンの howto が欲しい。

## Persona B — 前田 晋也（ロースキル・男性・35 歳）

### 背景

小売業の社内システム担当 8 年目。POS レジシステムの改修経験あり。ポイント管理の業務知識あり。

### 作業シナリオ

1. `point_ledger(id, user_id, amount, type, expires_at, created_at)` テーブルで設計。
   業務知識から「元帳（ledger）方式」を自然に選択。
2. 残高計算: `SELECT SUM(amount) FROM point_ledger WHERE user_id=? AND (expires_at IS NULL OR expires_at > now())`
3. ポイント利用はトランザクション内で残高チェック + INSERT（`amount` 負値でマイナス記録）。
4. クーポンは `coupons` + `coupon_usages(coupon_id, user_id, order_id)` で制限管理。
5. 割引計算: `discount_type = 'fixed'` なら `value` を直接引く、`'percent'` なら `order_amount * value / 100`。

### ハマりポイント

- **FIFO ポイント消費**: 有効期限が近いポイントから消費する「FIFO」ロジックが複雑。
  今回は「有効分を一括 SUM してまとめて消費」という簡易実装にした。
- **クーポン有効期限切れのエラーコード**: 「期限切れ」「条件未達」「重複使用」それぞれの
  エラーコードをどう設計するか（RFC 9457 Problem Details の適用方法）。
- **ポイント付与タイミング**: 注文時に付与するのか、配送完了時に付与するのかの判断ができない
  （今回はシンプルに即時付与）。

### 解決策 & 感想

業務知識でスムーズに実装できた。Problem Details の適用は `password-auth-argon2id.md` を参考にした。

> 「ledger 方式はすんなり設計できた。でも FIFO 消費は難しいな。
>  クーポンのエラーコード、Problem Details ってどう使うの？って最初分からなかったけど
>  howto 読んだら分かった。」

### DX スコア: ⭐⭐⭐（3/5）

良好に完成。FIFO ポイント消費と Problem Details 活用例が改善点。

## Persona C — 杉山 直美（シニア・女性・38 歳）

### 背景

フィンテック系スタートアップのバックエンドリード。「金銭に関するロジックは絶対に正確に」という信念。

### 作業シナリオ

1. `point_ledger` で設計。`amount` は符号付き整数（付与=正、利用=負）。
2. FIFO 消費を実装: 有効期限順に残高を持つポイントレコードをループで消費し、
   消費分を新しい負値レコードとして追記（append-only 維持）。
3. クーポン: `coupons` + `coupon_usages(UNIQUE(coupon_id, user_id))`。
   同一ユーザーの重複適用は DB 制約でブロック。
4. 割引計算を `DiscountCalculator` として UseCase から独立したサービスに。
   最大割引額のキャップ（例: 割引は注文額の 50% 以内）も実装。
5. OpenAPI で全エラーケース（期限切れ/条件未達/重複/ポイント残高不足）を定義。

### ハマりポイント

- **FIFO 消費の append-only 性**: 消費記録を「古いレコードを更新」せず「負値レコードを追加」
  する設計にしたが、これが正しいかどうか迷った。
- **整数演算の精度**: PHP の `int` 計算でポイント計算は問題ないが、
  パーセンテージ割引で `int` vs `float` をどう扱うか（整数除算の切り捨て方針）。
- **トランザクション内の FIFO ループ**: 多数のポイントレコードをループして消費する処理の
  パフォーマンスと、SQLite のロック時間への影響。

### 解決策 & 感想

高品質で完成。整数演算は `intdiv()` + 切り捨てポリシーで一貫。

> 「append-only でポイント消費を管理するのはイベントソーシングと同じ考え方で
>  NENE2 の event-sourcing howto が参考になった。
>  FIFO 消費の実装パターンを howto で示してくれると嬉しい。」

### DX スコア: ⭐⭐⭐⭐（4/5）

高品質完成。FIFO 消費と整数演算ポリシーの説明が改善余地。
