# DX Scenario 02: 問い合わせフォーム

## アプリ概要

カスタムフィールド定義・受付・通知を備えた問い合わせ管理 API。

| 機能 | エンドポイント例 |
|------|----------------|
| フォーム定義管理 | `GET /forms`, `POST /forms`, `GET /forms/{id}` |
| フィールド定義 | `POST /forms/{id}/fields`, `DELETE /forms/{id}/fields/{fid}` |
| 問い合わせ受付 | `POST /forms/{id}/submissions` |
| 送信内容一覧 | `GET /forms/{id}/submissions`, `GET /forms/{id}/submissions/{sid}` |
| 通知設定 | `PATCH /forms/{id}/notification`（メール宛先・件名テンプレート） |
| ステータス管理 | `PATCH /submissions/{id}/status`（未対応→対応中→完了） |

ポイント: 動的フィールド定義の JSON 保存、ステータス遷移、通知フック。

---

## Persona A — 伊藤 さくら（新卒・女性・24 歳）

### 背景

デザイン系専門学校卒後、エンジニアに転向して 1 年。HTML/CSS は得意で PHP は独学。
「フォームを作るだけ」のはずがバックエンド設計が難しい。

### 作業シナリオ

1. `forms` テーブルと `form_fields` テーブルを作るが、fields の `type` カラム（text/email/select 等）を
   どう扱うか分からず `TEXT NOT NULL` で全部受けることにする。
2. `POST /forms/{id}/submissions` のバリデーションで「動的フィールドを動的にバリデートする」
   方法に詰まる。Laravel の `Request::validate()` のような機能を探すが見当たらない。
3. バリデーション処理をコントローラーに 100 行以上書いてしまう。
4. 通知設定の「メールを送る」部分を実装しようとして、NENE2 がメール送信機能を持たないことを知る。
   外部ライブラリを使う方法を調べ始める（この時点で時間切れ）。
5. ステータス遷移は単純なカラム更新として実装。遷移ルールの検証なし。

### ハマりポイント

- **動的バリデーション**: フィールド定義に基づいてバリデーションを組み立てるパターンが howto にない。
- **メール送信**: フレームワーク外の機能をどう組み込むか全く分からない。
- **状態遷移ルール**: `state-machine-workflow-api.md` の存在を知らず、単純 UPDATE で実装。

### 解決策 & 感想

先輩に「動的バリデーションは UseCase 内でフィールド定義を読んでから検証する」と教わった。
メールは `symfony/mailer` を `composer require` して `NotificationService` を作る方法を知った。

> 「バリデーションが一番難しかった。
>  '動的に' っていうのが頭の中でイメージできなかったし、
>  フレームワークにメール機能がないのも最初びっくりした。」

### DX スコア: ⭐⭐（2/5）

動的フィールドとメール統合はドキュメントサポートが薄く、新卒には厳しい。

## Persona B — 橋本 龍二（ロースキル・男性・33 歳）

### 背景

受託 Web 制作 8 年目。WordPress のカスタム投稿タイプで似たようなものを作ったことがある。
「PHP は書けるが設計は苦手」。納期重視。

### 作業シナリオ

1. フォーム定義は `forms` テーブルの `fields_json` カラムに JSON 文字列で丸ごと保存する設計を選ぶ。
   （`form_fields` テーブルは作らない）
2. `POST /forms/{id}/submissions` の受付はフィールド定義を JSON デコードして foreach でバリデート。
   一応動作する。
3. 通知は PHP の `mail()` 関数を UseCase 内に直書き。テスト不可能な設計になる。
4. ステータス管理は `status` カラムを直接 UPDATE。遷移不正チェックなし（`processing→open` が可能な状態）。
5. `GET /forms/{id}/submissions` にページネーションをつけ忘れ、大量データで遅くなると後で発覚。

### ハマりポイント

- **JSON カラム vs 正規化テーブル**: 手早い JSON カラム設計の落とし穴（検索・バリデーション制限）に
  後で気づく。
- **`mail()` 直書き**: テスタビリティを考慮しない実装が後でテスト追加時に問題になる。
- **ステータス遷移ルール**: 不正遷移を防ぐガード処理を入れていないバグが本番で発覚。

### 解決策 & 感想

「動けばいい」で一度は完成したが、コードレビューで設計問題を複数指摘された。
`docs/howto/state-machine-workflow-api.md` を読んで遷移ロジックを書き直した。

> 「JSON カラムで手早くいけると思ったんだけど、後でバリデーションが
>  めちゃくちゃ面倒になった。あの時ちゃんとテーブル分けとけばよかった。」

### DX スコア: ⭐⭐⭐（3/5）

動作するが技術的負債を抱えて完成。リファクタリングで挽回できる。

## Persona C — 中村 典子（シニア・女性・41 歳）

### 背景

大手 SI 出身、現在スタートアップ CTOo。Java/Spring Boot から PHP に移ってきた。
「インターフェース設計と依存性逆転は当たり前」という感覚を持つ。

### 作業シナリオ

1. まず機能を洗い出してドメインモデルを設計:
   - `Form` / `FormField` / `Submission` / `SubmissionValue` / `NotificationConfig`
2. `FormRepository` / `SubmissionRepository` を作り、`SubmissionService` で
   動的バリデーションロジックを実装。
3. 通知は `NotificationPort` インターフェースを定義し、`MailNotificationAdapter` を実装。
   `NotificationPort` をコンテナにバインドして UseCase から使う。
4. ステータス遷移は `Submission::allowedTransitions()` メソッドで管理。
5. OpenAPI スキーマを書き、`composer check` で全チェックを通してから PR 作成。

### ハマりポイント

- **Port/Adapter パターンの配置**: `src/Notification/` を作るか `src/Integration/` にするか
  ディレクトリ規約が不明。
- **コンテナバインディング**: `ServiceProviderInterface` の追加方法を `src/DependencyInjection/` で
  確認する必要があり、少し時間がかかる。
- **動的バリデーション**: NENE2 の `Validation/` が静的ルールベースのため、
  動的バリデーションを UseCase 内で独自実装する必要がある。

### 解決策 & 感想

`src/` をシームレスに読めたため、Port/Adapter パターンを自然に組み込めた。
コンテナの使い方は `src/DependencyInjection/` のコード例で理解できた。

> 「PSR 準拠でインターフェース設計できているのは好感。
>  Notification みたいなインフラ横断機能のサービスプロバイダーの書き方が
>  もう少しドキュメント化されていると嬉しかった。」

### DX スコア: ⭐⭐⭐⭐（4/5）

品質高く完成。インフラ統合サービスプロバイダーのドキュメントが改善余地。

3. **JSON カラム vs 正規化テーブルの選択基準** — トレードオフの howto ドキュメントがあると良い。
