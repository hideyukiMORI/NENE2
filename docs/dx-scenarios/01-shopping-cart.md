# DX Scenario 01: ショッピングカート

## アプリ概要

EC サイトの購買フロー全体をカバーする中規模 API。

| 機能 | エンドポイント例 |
|------|----------------|
| 商品一覧・詳細 | `GET /products`, `GET /products/{id}` |
| カート操作 | `POST /cart/items`, `PATCH /cart/items/{id}`, `DELETE /cart/items/{id}` |
| 購入確定 | `POST /orders` |
| コンバージョン記録 | `POST /conversions`（購入完了イベント保存） |
| 注文履歴 | `GET /orders`, `GET /orders/{id}` |

ポイント: カート→注文の原子的変換、在庫デクリメント、コンバージョンログ。

---

## Persona A — 田中 勇樹（新卒・男性・23 歳）

### 背景

情報工学科卒、今年 4 月入社。学校の課題では Laravel を触ったが NENE2 は初めて。
「フレームワークはドキュメントを読めばなんとかなる」という楽観的な姿勢。

### 作業シナリオ

1. `docs/howto/add-database-endpoint.md` を開き、テーブル定義とリポジトリのパターンをコピー。
2. `products` テーブルは問題なく作れたが、カートテーブルをどこに置くか迷う。
   - `carts` + `cart_items` の 2 テーブル構成に気づくのに 30 分かかる。
3. `POST /cart/items` のバリデーションで「在庫チェックはコントローラーでやるのかユースケースでやるのか」に迷い、
   コントローラーに直接 SQL を書いてしまう。
4. `POST /orders` で「カートの中身を注文テーブルに移す＋在庫を減らす」を 2 回 INSERT に分けて書いてしまい、
   トランザクションの存在を先輩に指摘されて初めて知る。
5. `POST /conversions` は「なんのために必要か」が理解できず後回しにして未完成のまま提出。

### ハマりポイント

- **2 テーブル構造の設計**: `add-database-endpoint.md` は単一テーブルの例しかなく、
  リレーションのある設計への橋渡しが見つからない。
- **Business logic の置き場所**: コントローラー VS ユースケースの分離が初見では分からない。
- **トランザクション**: `DatabaseTransactionManagerInterface` の存在に気づかない。
- **コンバージョン概念**: ビジネス用語の背景知識がないと機能の意義が見えない。

### 解決策 & 感想

先輩に `docs/howto/` の `threaded-comments-api.md`（リポジトリパターンの良例）を紹介してもらい、
UseCase 分離と `tx->run()` の使い方を学んだ。

> 「ドキュメントにリレーション設計の例が 1 個あるだけでだいぶ違った。
>  あと "コンバージョン" って言葉、最初意味わかんなかった。」

### DX スコア: ⭐⭐⭐（3/5）

設計サポートが弱い。単一テーブル→マルチテーブルへの橋渡しドキュメントが欲しい。

## Persona B — 鈴木 めぐみ（ロースキル・女性・29 歳）

### 背景

Web 制作会社で 5 年。jQuery + PHP (手続き型) で動くものを作ってきた。
REST API は「なんとなく作れる」レベル。オブジェクト指向は苦手意識がある。

### 作業シナリオ

1. 既存プロジェクトの `products` エンドポイントを参考にテーブルとリポジトリを作る。
2. カート操作は `cart_items` テーブルに直接 UPSERT して実装。`carts` テーブルは作らない判断。
3. `POST /orders` を実装するとき、「カートを空にする」処理と「注文を作る」処理を
   別々のリクエストにしてしまう（フロントで 2 回叩く設計）。
4. `DELETE /cart/items/{id}` で `404 vs 403` の返し分けに悩み、とりあえず全部 400 で返す。
5. コンバージョン記録は `orders` テーブルに `converted=1` カラムを足して済ませる。

### ハマりポイント

- **2 ステップ→1 アトミック操作**: 「2 回に分けてフロントで叩けばいい」発想から抜け出せない。
- **エラーコード設計**: 404/403/409 の使い分けを知らない。`docs/howto/` に
  エラーコード一覧があることを後で知る。
- **コンバージョンのモデリング**: 「フラグでいい」思考。独立イベントログとして設計する発想がない。

### 解決策 & 感想

`docs/howto/event-sourcing-cqrs-api.md` のコンバージョンログパターンを読んで「なるほど」。
ただし「こういうドキュメントがある」と誰かに教えてもらわないと辿り着けなかった。

> 「エラーコードって RFC があるの初めて知った。
>  Problem Details、名前かっこいいと思ったけど最初は全然意味わからんかった。」

### DX スコア: ⭐⭐⭐（3/5）

動くものは作れる。品質は低め。howto の発見性が課題。

## Persona C — 山本 健一（シニア・男性・38 歳）

### 背景

Laravel + Symfony 経験 10 年。最近 Go での API 開発もしている。
「ミニマルフレームワークはどうせ貧弱」という先入観あり。性能・型安全性を重視。

### 作業シナリオ

1. まず `src/` を一通り読む。PSR 準拠・DI・トランザクション境界を確認してひとまず安心。
2. `ProductRepository` / `CartRepository` / `OrderRepository` を一気に設計し、
   UseCase にビジネスロジックを集約する方針を決める。
3. 在庫デクリメントは `SELECT ... FOR UPDATE` を使いたいが、SQLite では使えないと気づく。
   MySQL モードで試そうとして `docker-compose.yml` を読み始める。
4. `POST /orders` の UseCase が長くなるため、`OrderService` を UseCase と Repository の間に置きたいが、
   NENE2 のアーキテクチャ推奨パターンが不明で一旦 UseCase に全部書く。
5. OpenAPI スキーマを先に書き、`composer openapi` で契約テストを通してから実装する TDD スタイルで進める。

### ハマりポイント

- **SELECT FOR UPDATE / 悲観ロック**: SQLite 前提の howto では言及がなく、
  MySQL 移行コストを自分で調べる必要がある。
- **Service 層の有無**: UseCase と Repository の間に Service を置く公式推奨が見当たらない。
- **OpenAPI 先行開発**: `composer openapi` が強力だが、スキーマを書いてから実装に入る
  ワークフロー例がドキュメントにない。

### 解決策 & 感想

`docs/adr/` を読んでアーキテクチャ判断の根拠を確認。ADR に Service 層を不要とした記録はなかったが、
UseCase に収める例が多いことから「NENE2 の流儀に合わせる」判断をした。

> 「悲観ロックの howto があれば嬉しい。あとテスト先行の開発フロー例が欲しい。
>  コードは読めばわかるけど、文化・流儀の学習コストが思ったより高かった。」

### DX スコア: ⭐⭐⭐⭐（4/5）

品質高く完成できる。MySQL 固有機能とサービス層の公式見解があればさらに良い。

**共通のフリクション**:
1. **マルチテーブル設計の howto がない** — 単一テーブル例から複数テーブルへの橋渡しドキュメントが欲しい。
2. **コンバージョン・イベントログの概念説明** — ビジネス用途の説明があると低スキルペルソナが迷わない。
3. **トランザクション利用例が howto で薄い** — `tx->run()` の使い方を明示した howto リンクがあると良い。
