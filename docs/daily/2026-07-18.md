## U-8: gen:feature に mutation archetype（4値 union）を追加（Issue #1579 / PR #1580・Merge）

- 背景: `gen:feature` は read/query archetype（3値 `loading|error(retry)|success`・empty は success 導出）のみを生成し、form/mutation の canonical union が未定だった〔実測: plopfile / テンプレ〕。
- 事実訂正（会議前提の裏取り・設計ノートで既報）: 「payout に4値形の現物」は正確には **list feature の4値**（`loading|empty|error|success`・`empty` 明示 arm）で、form/mutation は react-query 生 boolean を View で直接消費＝canonical union 不在だった〔実測: payout frontend の grep〕。
- 実装: `gen:feature <verb-noun> <noun> --mutation` で4値 union `idle | submitting | error(retry) | success(reset)`。flag は `process.argv` 検出・既定 query 非破壊。error 腕の retry は同一 input 再 mutate・success 腕に reset(idle 復帰) 同梱〔指揮裁定 Q2・伝聞〕。遷移テストテンプレ4本〔実測〕。
- 素振り発見の latent defect も同梱修正: entity `mutations.ts` テンプレの `UseMutationResult` 第3型引数 `unknown` が既定値で `no-unnecessary-type-arguments` に触れ、`gen:entity --write` 出力が lint 落ちしていた（未 commit ゆえ CI 未検出）〔実測: eslint 出力〕。
- 設計判断: 3案（tools(?callable) / fromTools / interface 抽出…は MCP 側の話。本件は archetype 形の3案）は独断せず指揮へ fork を上げ (C) 相当の裁定を得てから実装〔伝聞: hub 裁定〕。
- 検証: 実 gen 素振り（gen:entity payment y ＋ gen:feature submit-payment payment --mutation）で type-check / eslint --max-warnings 0 / 生成テスト4本 いずれも緑〔実測〕。`npm run check` 28 tests・CI 3本 SUCCESS〔実測〕。
- merge: 指揮の必読レビュー後 squash merge・main 同期〔実測: SHA `c2b5311`〕。
- Issue 駆動の逸脱を1件自己是正: ローカル main への直コミット＋コミット番号誤り（#1570）を push 前に feature ブランチ退避・main 復旧・#1579 へ訂正〔実測: git〕。

## mutation exemplar（submit-note）で恒久 CI ゲート化（Issue #1581 / PR #1582・Merge）

- 目的: #1580 の生成物は素振り検証のみで恒久 CI ゲートが無かった。query の view-notes exemplar と対称に committed exemplar を追加し、CI が type-check/lint/test を恒久的に守る。
- 設計判断: exemplar の忠実度に本質的分岐（生 gen faithful vs polished・note 再利用時の endpoint 整合）があり、(A)/(B)/(C) を理由つきで指揮へ上げ **(C)** 裁定〔伝聞: hub〕。決め手は view-notes が既に (C) 型（feature=生 gen byte一致・entity=実 endpoint fleshed）〔実測: use-view-notes.ts はテンプレとバイト一致〕。
- 実装: feature `submit-note` は生 gen 出力を commit（hook/view byte一致・test のみ POST endpoint を実 API `/api/examples/notes` へ置換＋ドメイン assertion）。note entity に create 側を fleshed 追加（`useCreateNote` → POST `/api/examples/notes`）＋由来コメント明記（生テンプレとの差分は endpoint のみ・input 型付けは consumer TODO のまま）〔実測・指揮条件〕。
- page/route 配線は不要と実測確定（未配線でも lint --max-warnings 0 通過＝最小スコープ）〔実測〕。
- 検証: `npm run check` 32 tests 緑・CI 3本 SUCCESS〔実測〕。merge: SHA `2e70d35`〔実測〕。

## 命名衝突 fail-fast ガード（Issue #1583 / PR #1584・Merge）

- 発見: exemplar 制作中、`gen:feature create-note note --mutation` が feature hook `use{CreateNote}` = `useCreateNote` と entity `useCreate{Note}` = `useCreateNote` の**同名衝突**で自己再帰（Maximum call stack size exceeded）＋ import 衝突を起こすと判明〔実測: tsc / vitest〕。「新規作成 feature を `create-<noun>` と呼ぶ」最も自然な命名で発火する。
- 対処〔指揮裁定・伝聞〕: plopfile に fail-fast ガード。`pascalCase(verbNoun) === "Create"+pascalCase(noun)` を検出したらエラー停止＋代替 verb 提案（submit-/register-/add-）。entity 側 `useCreate<Noun>` は ST-5 固定命名につき不動。
- 検証: 決定性テストに衝突ケース追加（非ゼロ終了・ファイル未生成・提案メッセージ内容まで固定）。`npm run check` 33 tests 緑・CI 3本 SUCCESS〔実測〕。merge: SHA `2a74ad3`〔実測〕。
